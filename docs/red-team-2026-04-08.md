### [HIGH] Silent data loss — `beforeUnload` skipped when PATCH succeeds mid-debounce

**File:** `frontend/src/features/editor/hooks/useEditorAutosave.ts:97–120`

**Attack path:**
1. User edits state A → fingerprint effect fires → `setIsDirty(true)`, debounce timer starts (800 ms)
2. Debounce fires → PATCH for A is in flight
3. User immediately makes edit B (PATCH still in flight)
4. PATCH for A succeeds → `onSuccess` runs: `baselinePersistFingerprintRef.current = fingerprint(CURRENT STATE = B)`, `setIsDirty(false)`
5. `isDirtyRef.current` is now `false` — the fingerprint effect on next render sees `nextFp === baseline` (both = B) → no new save or dirty flag
6. User closes tab within the 800 ms debounce window for B
7. `beforeUnload` handler: `!isDirtyRef.current` → returns early, no emergency flush
8. Edit B is lost silently

**Impact:** User loses all work done between when A's PATCH succeeds and when the browser is closed. No error, no warning.

**Why it works:** `onSuccess` sets `baselinePersistFingerprintRef` to `editorPublishStateRef.current` — the *latest* snapshot — not the snapshot that was actually saved. When B exists in the current snapshot, the baseline is advanced to include B, which removes the fingerprint difference that would have triggered a resave. The `isDirty = false` call then clears the UI flag before B's debounce fires.

**Fix:** Capture the fingerprint at the moment the PATCH payload is built, not inside `onSuccess`:

```typescript
// Before calling mutate/queueSave:
const sentFingerprint = persistFingerprint(snap);

// In onSuccess:
baselinePersistFingerprintRef.current = sentFingerprint; // not editorPublishStateRef.current
```

Alternatively: only call `setIsDirty(false)` in `onSuccess` if the current fingerprint still equals `sentFingerprint`. If the user has already made new edits, leave `isDirty = true`.

### [MEDIUM] `probeHasAudioStream` has no timeout — export worker stalls indefinitely

**File:** `backend/src/domain/editor/run-export-job.ts:34–57`

```typescript
const proc = Bun.spawn(["ffprobe", ..., filePath], { stdout: "pipe", stderr: "ignore" });
await proc.exited;  // no timeout
```

**Attack path:** Upload a specially crafted media file that causes `ffprobe` to hang (e.g., a malformed container with no valid stream headers, triggering a parse loop). The `probeHasAudioStream` call awaits `proc.exited` with no deadline. The entire export job hangs. The Redis lock (`runExportJob` acquires one per composition version) is held for the duration.

**Impact:** The user's export worker slot is permanently occupied. If the worker pool is small, a handful of crafted uploads could exhaust the export capacity (DoS). The Redis lock means no re-queue is possible until the process is manually killed or the server restarts.

**Why it works:** `Bun.spawn` does not support a timeout option natively; `proc.exited` is a promise with no built-in deadline. `stderr: "ignore"` also suppresses any ffprobe error output, making the hang silent in logs.

**Fix:** Race `proc.exited` against a deadline. Kill the process on timeout:

```typescript
const timeoutMs = 10_000;
const timeout = new Promise<never>((_, reject) =>
  setTimeout(() => { proc.kill(); reject(new Error("ffprobe timeout")); }, timeoutMs)
);
await Promise.race([proc.exited, timeout]);
```

### [LOW] `adelay` filter injection via NaN/Infinity — export job failure

**File:** `backend/src/domain/editor/run-export-job.ts` (audio filter construction)

```typescript
const delayMs = Math.max(0, Math.round(clip.startMs ?? 0));
filterParts.push(
  `[${inputIdx}:a]atrim=...adelay=${delayMs}|${delayMs}[a${i}]`,
);
```

**Attack path:** Persist a clip with `startMs: null` works fine (defaults to 0). But `startMs: Infinity` or `startMs: NaN` reaches here because `Math.max(0, NaN) = NaN` and `Math.round(Infinity) = Infinity`. The filter string becomes `adelay=NaN|NaN` or `adelay=Infinity|Infinity`. ffmpeg rejects this with a filter-parsing error, killing the entire export job.

The same risk exists for `trimStartMs` → `trimStart` and `durationMs` → `sourceDurationSec`, which are also interpolated directly:

```typescript
`[${i}:v]trim=start=${trimStart}:duration=${sourceDurationSec}`
```

**Impact:** An attacker who can write arbitrary clip metadata (e.g., via a crafted autosave PATCH that bypasses Zod validation) can reliably abort their own export job. Not RCE — ffmpeg is invoked with an array, no shell. Not cross-user. This is a per-user self-DoS.

**Why it works:** The Zod schema for clip fields likely uses `z.number()` which accepts `NaN` and `Infinity` unless `.finite()` is added.

**Fix:** Add `.finite()` to numeric clip fields in the Zod schema:

```typescript
startMs: z.number().finite().nonnegative(),
durationMs: z.number().finite().positive(),
trimStartMs: z.number().finite().nonnegative().optional(),
```

And defensively guard the filter construction:

```typescript
const delayMs = Number.isFinite(clip.startMs) ? Math.max(0, Math.round(clip.startMs)) : 0;
```

### [LOW] `alignMediaClipSourceBounds` falsy `assetId` check skips enforcement for empty string

**File:** `backend/src/domain/editor/timeline/track-overlaps.ts:194–202`

```typescript
if (
  !clip.assetId ||  // falsy: null, undefined, ""
  !Number.isFinite(sourceMaxDurationMs) || ...
) {
  return clip;  // no bounds enforcement
}
```

**Attack path:** A clip with `assetId: ""` (empty string — technically valid JSON, falsy in JS) bypasses source-bounds alignment. The clip's `durationMs`/`trimEndMs` are not clamped to `sourceMaxDurationMs`. On the frontend, this means a clip with no actual asset can have an unbounded duration, which would be passed to ffmpeg as a filter duration. Downstream, `[${i}:v]trim=start=0:duration=${LARGE_NUMBER}` is valid ffmpeg syntax and will produce a long silent-black video segment rather than failing.

**Impact:** Low — the result is a corrupted export (extended silent/black), not arbitrary code execution. Requires writing `assetId: ""` to the DB, which the API's Zod schema may prevent.

**Fix:** Use explicit null/undefined check:

```typescript
if (
  clip.assetId == null || clip.assetId === "" ||
  ...
```

Or better: change the type to only accept truthy strings and validate at the API boundary.

## Bug: Video clips do not play — `usePreviewMediaSync.ts`

> **Severity: Critical (regression introduced in this diff)**  
> **Symptom:** Pressing play does nothing. Video clips only update their displayed frame when toggling pause — the `currentTime` seek runs, but `element.play()` is never called.

### Root Cause — Single Removed Line

**Commit 364826c → staged diff, `usePreviewMediaSync.ts` line ~117:**

```diff
-        syncElement();
         if (isPlaying && requestFrame) {
```

`syncElement()` was previously called **unconditionally** — every sync cycle — before the `requestVideoFrameCallback` block. That call is what started playback (`element.play()`) and seeked the element. The rVFC callback was then registered to handle *future* frame-accurate position corrections.

The staged diff removed that unconditional call. Now `syncElement()` is only called **from inside** the rVFC callback (when playing) or the `else` branch (when not playing):

```typescript
// Current code — the bug
if (isPlaying && requestFrame) {
  // ...cancel existing handle...
  const handle = requestFrame(() => syncElement());  // syncElement ONLY from here
  videoFrameHandlesRef.current.set(clip.id, handle);
} else {
  syncElement();  // only runs when NOT playing
}
```

### Why `requestVideoFrameCallback` Never Fires

`requestVideoFrameCallback` fires when the browser presents a new decoded video frame to the compositor. A paused video presents no new frames — so the callback never fires. Classic deadlock:

```
element.play() → needs rVFC to fire
rVFC fires     → needs element to be playing
```

The video stays paused forever once playback is initiated.

### Why the Bug Manifests as "Frame Changes on Pause"

When `isPlaying` becomes `false`:
- The `else { syncElement(); }` branch runs directly
- `syncElement()` executes `element.currentTime = targetTime` (the seek threshold check passes on first run)
- Seeking a paused video updates the displayed frame → the user sees the frame snap to the current playhead position

This perfectly matches the reported symptom.

### Compounding Factor: rVFC Handles Cancelled Every Frame

The `onPublishCurrentTime` throttle (150 ms) was also removed in `usePreviewPlaybackBridge.ts`:

```diff
- if (Math.abs(clampedTimeMs - lastPublishedAtRef.current) >= PLAYHEAD_PUBLISH_INTERVAL_MS) {
-   lastPublishedAtRef.current = clampedTimeMs;
-   onPublishCurrentTime(clampedTimeMs);
- }
+ onPublishCurrentTime(clampedTimeMs);
```

`onPublishCurrentTime` dispatches `SET_CURRENT_TIME` to the reducer. `usePreviewMediaSync` has `currentTimeMs` in its effect dependency array, so it now re-runs on every rAF tick (~60 fps). Each run's cleanup cancels all rVFC handles before they fire. This means even if rVFC somehow started, it would be cancelled within 16 ms by the next tick's cleanup. The rVFC path is entirely dead under the current architecture.

### Fix

**Minimal fix** — restore the unconditional `syncElement()` call before the rVFC block in `usePreviewMediaSync.ts`:

```typescript
if (isPlaying && requestFrame) {
  const cancelFrame = ...;
  const existingHandle = videoFrameHandlesRef.current.get(clip.id);
  if (existingHandle !== undefined && cancelFrame) {
    cancelFrame(existingHandle);
  }
  syncElement();  // ← RESTORE: starts play() immediately; rVFC handles future corrections
  const handle = requestFrame(() => syncElement());
  videoFrameHandlesRef.current.set(clip.id, handle);
} else {
  syncElement();
}
```

**Secondary fix** — restore the 150 ms publish throttle in `usePreviewPlaybackBridge.ts` so `usePreviewMediaSync`'s effect doesn't re-run at 60 fps. This prevents rVFC handle churn and reduces unnecessary re-renders. If a full-rate reducer dispatch is intentional for timeline timecode accuracy, a separate lightweight subscribe path for the media sync (bypassing React re-render) would be the right architecture.

## `load-fixtures.ts` — `estimateMp3DurationMs`

**Scope:** `backend/src/services/video-generation/dev-fixtures/load-fixtures.ts`

### Threat model

Despite the `dev-fixtures/` directory name, `estimateMp3DurationMs` is imported and called in **production code** at `backend/src/services/tts/elevenlabs.ts:50,113`. The buffer it parses is the raw HTTP response body from the ElevenLabs API. ElevenLabs is a trusted third party — an attacker cannot normally influence the content of that response. The risk surface is narrow but worth documenting.

---

### [LOW] ID3v2 syncsafe integers not masked — wrong cursor on malformed headers

**File:** `load-fixtures.ts:84–88`

```typescript
const size =
  ((buffer[6] ?? 0) << 21) |
  ((buffer[7] ?? 0) << 14) |
  ((buffer[8] ?? 0) << 7) |
  (buffer[9] ?? 0);
```

**What's wrong:** The ID3v2 spec (section 3.1) defines the tag size as four "syncsafe integers" where only the lower 7 bits of each byte are significant (MSB is always 0). The correct decode is:

```typescript
((buffer[6] & 0x7F) << 21) | ((buffer[7] & 0x7F) << 14) | ((buffer[8] & 0x7F) << 7) | (buffer[9] & 0x7F)
```

Without the `& 0x7F` mask, a tag with any MSB set in the size bytes produces a cursor value up to ~539 MB. When the cursor exceeds `buffer.length`, the while loop immediately exits with `frameCount = 0`, triggering the byte-size heuristic fallback.

**Impact:** The returned `durationMs` is wrong — the heuristic assumes ~128 kbps and overestimates for TTS audio (ElevenLabs returns 128 kbps MP3, so the heuristic happens to be accurate here). The practical impact is low in the current setup, but any shift in ElevenLabs output bitrate (e.g., upgrading to a higher-quality model) would cause the heuristic to return wrong durations for files with the MSB set in their ID3 size bytes.

**Exploitability:** Only reachable if ElevenLabs returns an MP3 whose ID3 tag size bytes have the MSB set — a malformed tag by the spec. Normal encoders don't produce this. Not exploitable against the current ElevenLabs endpoint.

**Fix:** Add the `& 0x7F` mask to each byte:

```typescript
const size =
  ((buffer[6]! & 0x7F) << 21) |
  ((buffer[7]! & 0x7F) << 14) |
  ((buffer[8]! & 0x7F) << 7) |
  (buffer[9]! & 0x7F);
```

### [LOW] Production function lives in `dev-fixtures/` — silent coupling

**File:** `load-fixtures.ts` → imported from `elevenlabs.ts:6`

`estimateMp3DurationMs` is a production-critical function (it determines every voiceover clip's `durationMs` which drives timeline layout and export trim values) but is defined inside `dev-fixtures/`. A developer auditing or refactoring the dev fixtures path has no obvious signal that this function is on the production hot path.

**Exploit path:** Not a runtime security issue. The risk is a future developer removing or breaking this file in a "dev only" cleanup. `durationMs = 0` or `NaN` from a broken parser would persist into the DB and likely cause FFmpeg export failures (zero-duration audio trim).

**Fix:** Move `estimateMp3DurationMs` (and `estimateMp3DurationMsFromBufferSize`) to `backend/src/services/tts/mp3-duration.ts` or similar. Update imports in `elevenlabs.ts` and the test file.

### [INFO] `b3` is read but never used

**File:** `load-fixtures.ts:110`

```typescript
const b3 = buffer[cursor + 3] ?? 0;
```

`b3` is read from the buffer but not referenced anywhere. The 4th header byte encodes channel mode, mode extension, copyright, original, and emphasis flags — none of which affect frame length or duration. This is dead code; the read is harmless but should be removed.

## Hardening Ideas

- **`useEditorAutosave`**: Add a `saveTimerRef.current != null` check inside `beforeUnload` as a secondary safeguard — if a debounce is pending, always flush regardless of `isDirty`.
- **Export job**: Add a per-clip numeric guard function (checks for finite, non-negative) and sanitize all user-provided numeric values before filter string construction. This is one function, called for every clip field that goes into a filter.
- **`probeHasAudioStream`**: Log `proc.exitCode` when non-zero so silent probe failures are observable in export logs.
- **`usePreviewPlaybackBridge`**: Now dispatches `SET_CURRENT_TIME` on every rAF tick (removed the 150 ms throttle). This is intentional but worth profiling — the reducer allocates a new state object on every frame during playback, which may surface as GC pressure on long sessions.
